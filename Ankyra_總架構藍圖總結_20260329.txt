Ankyra 總架構藍圖總結
日期：2026-03-29

一、專案定位
Ankyra 定位為面向工作場景的多用戶 AI 助理平台。
平台核心價值不是單次聊天，而是：
- 多用戶與多助理運作
- 長期 conversation 與 memory 沉澱
- 外部信箱與工具整合
- 主動整理待辦、晨會摘要與提醒
- 將公司價值保留在自有 conversation ledger、memory 與分析系統中

- 計費／方案／使用量模型
- 稽核／日誌／治理模型

三、系統核心主線
1. 組織與使用者主線
- Organization -> User -> Membership -> OAuthIdentity

2. 助理產品主線
- AssistantDefinition -> AssistantVersion -> AssistantInstance

3. 對話主線
- AssistantInstance -> Conversation -> ConversationEvent -> ConversationSummary

4. 記憶主線
- User -> Memory -> MemoryEmbedding

5. 工具主線
- User/Organization -> ToolConnection -> ToolPolicy -> ToolInvocationLog

6. 排程與通知主線
- Schedule -> Job -> TaskSnapshot / BriefingSnapshot -> NotificationMessage

7. 治理主線
- Policy -> ApprovalRecord / PolicyChangeLog / AuditLog / SecurityEvent / RetentionRule

四、AI 執行主線
- 收到任務
- 載入 assistant instance / version
- 載入 user / organization / policy / memory / tool 上下文
- 組裝 AIContextPayload
- 判斷可用資料與可用工具
- 呼叫 OpenAI
- 進入 tool calling 迴圈
- 產出 assistant 回應
- 回寫 conversation、summary、memory、snapshot、audit、usage、monitoring

五、架構分層
1. 應用主資料層
- 使用者、助理、conversation ledger、memory、工具授權、通知、計費、稽核

2. AI 執行狀態層
- OpenAI Responses / Conversations API 執行狀態

3. 分析與知識層
- 待辦快照、摘要快照、使用成效、營運報表、AI 分析與治理分析

六、MVP 最小閉環
閉環 1
- OAuth2 登入
- 助理申請
- 對話持久化

閉環 2
- Gmail 或 Outlook 整合
- 今日待辦整理
- 晨會摘要生成

閉環 3
- LINE 或 Telegram 通知
- 每日提醒與晨會摘要推送

七、MVP 第一版建議範圍
- 支援 Google / Microsoft / WeChat / Twitter OAuth2，架構保留擴充
- 助理主力類型：
  - 郵件整理
  - 今日待辦
  - 晨會摘要
  - 會議整理
  - 專案追蹤
- 能力等級：
  - 初階
  - 進階
  - 專業
- 外部整合建議先做 Gmail 或 Outlook 擇一
- 通知建議先做 LINE 或 Telegram 擇一

八、主要原則
- OpenAI conversation 不是唯一真相
- 重要長期資料保留在 Ankyra
- 助理版本不覆蓋，只新增
- conversation 與 memory 分離
- tool 與 policy 先判定再執行
- 審批、稽核、retention 要從第一版就預留
