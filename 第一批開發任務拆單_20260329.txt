Ankyra 第一批開發任務拆單
日期：2026-03-29

一、目標
本文件將 MVP 實作規劃轉成可直接分派、估時與排程的第一批開發任務。
本輪以「可完成 MVP 最小閉環」為目標，不追求一次做完整商用版。

二、MVP 實作收斂版
本輪建議實作範圍：
- OAuth2：Google、Microsoft
- 助理類型：郵件整理、今日待辦、晨會摘要
- 能力等級：初階、進階
- 外部整合：Gmail 或 Outlook 擇一
- 通知通路：LINE 或 Telegram 擇一
- AI 能力：整理、摘要、分析、提醒
- 治理能力：approval、audit、policy 最小版

三、工作分組
1. 架構與專案底座
2. 身分與租戶
3. 助理目錄與實例
4. Conversation 與 AI orchestration
5. Connector 與外部資料整合
6. Snapshot 與通知
7. Approval / Audit / Policy
8. 管理端與基本前端頁面
9. 測試與驗收

七、MVP 最大風險
1. OAuth provider 過多導致前期卡住
建議：
- MVP 實作先落 Google / Microsoft
- WeChat / Twitter 保留 provider 架構與 interface

2. Connector 失敗率與 token 管理
建議：
- 只選一個信箱 provider 先落地
- 先做 read-only

3. AI 輸出品質不穩
建議：
- 先鎖定固定輸出格式與固定場景

4. Memory 過早複雜化
建議：
- 先做摘要型與偏好型 memory

5. 範圍膨脹
建議：
- 助理控制在 3 種
- 通知控制在 1 個通路
- provider 控制在 1 個信箱整合

八、第一批驗收條件
- 使用者可完成 OAuth2 登入
- 使用者可進入對應租戶
- 使用者可申請至少一種助理
- 使用者可建立 conversation 並獲得 AI 回應
- 系統可讀取單一信箱 provider 並整理郵件
- 系統可產出今日待辦與晨會摘要
- 系統可透過單一通知通路推送結果
- 管理者可查看 approval 與 audit 最小紀錄
